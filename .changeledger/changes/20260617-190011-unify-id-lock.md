## Request

El proyecto tiene dos implementaciones de lock de archivo que sirven a propósitos
similares pero con estrategias de staleness opuestas:

- `new.mjs:acquireIdLock` — lock PID-liveness: stale si el proceso owner ya no existe
- `atomic-write.mjs:mutateFileAtomic` — lock timeout: stale si el archivo tiene más de N ms

Son independientes en su scope pero pueden derivar en comportamientos inconsistentes
si alguna vez se unifican o si una de las dos introduce un bug que la otra no tiene.
Además, `new.mjs` ya tiene el ticket `20260617-190006` (spin safety) que modifica
`acquireIdLock` — la oportunidad de revisar si se pueden alinear estrategias.

## Proposal

Evaluar si las dos implementaciones deben converger o si sus diferencias son
intencionales e irreconciliables:

- `acquireIdLock` (new.mjs): protege contra colisión de id entre procesos concurrentes.
  PID-liveness es más robusto que timeout aquí (un agente lento no libera el lock
  prematuramente). Depende de `process.kill(pid, 0)`.
- `mutateFileAtomic` (atomic-write.mjs): protege contra escritura concurrente de un
  mismo archivo. Timeout es más simple y portable aquí.

Si las diferencias son intencionales (probable), documentarlas explícitamente para
que futuros mantenedores no intenten unificarlas por accidente. Si convergen, extraer
una primitiva de lock compartida con estrategia configurable.
