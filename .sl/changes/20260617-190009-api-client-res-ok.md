## Request

`getRepo` en `src/viewer/public/api.js` usa `res.text()` sin verificar `res.ok`.
Si el servidor retorna un error HTTP (404, 410, 500), el body de error se devuelve
como texto y el caller intenta parsearlo como JSON del repo, fallando silenciosamente
con un error de parse o datos incorrectos.

## Investigation

`src/viewer/public/api.js:3-6`:

```js
export const getRepo = async (project) => {
  const res = await fetch(`/api/repo?project=${encodeURIComponent(project)}`);
  return res.text();   // sin res.ok check
};
```

El servidor puede retornar:
- `404 { "error": "no project" }` — proyecto no encontrado
- `410 { "error": "project path is gone" }` — path borrado
- `500 { "error": "..." }` — error interno

El caller (`state.js` o similar) recibe un string de error y falla de forma
impredecible. Las otras funciones en `api.js` usan `.then(r => r.json())` tampoco
con `r.ok` check, pero al menos son GET simples que el servidor responde con 200
en éxito. `getRepo` es la única que puede genuinamente retornar 404/410.

Fix: lanzar un error con el status si `!res.ok`, o retornar `null`/un objeto de
error estructurado que el caller puede manejar.

No hay tests client-side para las respuestas de error de `api.js`.

## Specification

### CR1 — getRepo lanza si res.ok es false
- **Given** el servidor retorna 404 o 410 para `/api/repo`
- **When** el cliente llama `getRepo(project)`
- **Then** la función lanza un `Error` con el status HTTP en el mensaje (ej. `"HTTP 404"`)
- **And** no retorna el body de error como si fuera datos válidos

### CR2 — Caller recibe el error correctamente
- **Given** `getRepo` lanza
- **When** el caller tiene un try/catch o manejo de promise rejection
- **Then** puede distinguir "no project" de "project gone" de "server error" por el status

### CR3 — Tests cubren error responses de getRepo
- **Given** el servidor retorna 404
- **When** el cliente llama `getRepo`
- **Then** el test verifica que la promise es rejected (o que lanza)
