## Request

Buscar across todos los proyectos registrados desde el visor global, no solo dentro del proyecto activo. Resultados agrupados por proyecto.

## Investigation

- Hoy la búsqueda (`#search`) filtra solo los changes del proyecto cargado en el
  cliente. No hay forma de buscar en los demás repos sin cambiar de proyecto.
- El server ya resuelve la lista de proyectos (`resolveProjects`) y sabe cargar
  cualquiera (`loadRepo(p.path)`). Cada change ahora trae su `text` crudo, así
  que un match full-text server-side es directo.
- En modo local (`sl view .`) solo hay un proyecto: la búsqueda global coincide
  con la local; el toggle solo aporta con varios proyectos.

## Proposal

Endpoint `GET /api/search?q=` que recorre los proyectos vivos, hace match
full-text sobre el texto crudo de cada change y devuelve resultados agrupados por
proyecto: `[{ project:{id,name}, matches:[{id,title,type,status}] }]`.

- **Visor**: toggle "Global" junto a la búsqueda. Activo + query → consulta el
  endpoint y rinde una sección de resultados agrupados; clic en un resultado
  navega al proyecto+change (`gotoChange`). Sin query o toggle off → comportamiento
  actual (filtro local).

Descartado: cargar todos los repos en el cliente y filtrar ahí — escala mal y
duplica IO; el server ya tiene acceso a los paths.

## Specification

### CR1 — Buscar en todos los proyectos
- **Given** varios proyectos registrados con changes que casan una query
- **When** activo "Global" y escribo la query
- **Then** se listan los matches de todos los proyectos vivos, agrupados por proyecto

### CR2 — Navegar a un resultado
- **Given** resultados de búsqueda global
- **When** hago clic en un resultado de otro proyecto
- **Then** el visor cambia a ese proyecto y abre ese change

### CR3 — El endpoint agrupa y filtra
- **Given** una query
- **When** se llama `GET /api/search?q=`
- **Then** devuelve solo proyectos vivos con al menos un match, cada uno con sus matches
