## Request
The viewer HTTP endpoints rely on blocking I/O while handling requests. The
highest-impact path is `/api/repo`, which calls `loadRepo()` and reads every
change/spec file before responding. In a Node.js single-threaded server, large
histories can stall unrelated requests while that synchronous work runs.

## Proposal
Introduce async read entry points for the viewer data path instead of trying to
make every CLI command async at once. Keep the existing synchronous `loadRepo()`
for command code, add an async equivalent in `src/repo.mjs`, and make
`src/commands/view.mjs` await it for viewer API responses.

Static asset reads and vendor files can stay synchronous in this change because
they are small, bounded files; the purpose here is unblocking the large repo
history path.
