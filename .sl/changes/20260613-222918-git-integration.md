## Request

Vincular changes con git: detectar branch/commits/PR que referencian `[#id]` y mostrarlos en el detalle del change. Trazabilidad doc↔código.

## Investigation

- La convención de commits (AGENTS.md §6.4) ya referencia el id: `[#<id>]`. Eso
  da una señal sólida y local para enlazar commits ↔ change.
- Detectar commits: `git log --all -F --grep "[#<id>]"`. Detectar branches por
  nombre que contenga el id (la convención de branch no fija el id, así que suele
  venir vacío, pero es barato e inocuo).
- **PR**: detectarlos en vivo requiere red y `gh`/API autenticada. El visor es
  local y sin red: queda fuera de alcance (la traza por commit ya cubre doc↔código).
- El módulo debe ser testeable sin un repo real: inyectar el runner de git.

## Proposal

Módulo `git.mjs` con `gitRefs(repoRoot, id, run)`:

- `run(args)` ejecuta git y devuelve stdout (por defecto `execFileSync`); en test
  se inyecta un fake. Si el repo no es git o git falla → devuelve vacío.
- Devuelve `{ commits:[{sha,subject,date}], branches:[name] }`.

Endpoint `GET /api/git?project=&id=` resuelve el path del proyecto y llama a
`gitRefs`. El detalle del change pide esos refs y los muestra en una sección
**Git** (commits con sha corto + subject; branches).

Descartado: lookup de PR en vivo (red/`gh`, fuera del modelo local del visor);
parsear el id desde el cuerpo del commit más allá del marcador `[#id]`.

## Specification

### CR1 — Detectar commits que referencian el id
- **Given** commits cuyo mensaje contiene `[#<id>]`
- **When** se piden los refs git del change
- **Then** se listan esos commits (sha, subject, fecha)

### CR2 — Robustez fuera de git
- **Given** un path que no es repo git (o git falla)
- **When** se piden los refs
- **Then** se devuelve `{ commits:[], branches:[] }` sin lanzar

### CR3 — Mostrar en el detalle
- **Given** un change con commits enlazados
- **When** abro su detalle en el visor
- **Then** veo una sección Git con esos commits
