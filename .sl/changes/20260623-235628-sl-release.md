## Request

Comando `sl release` que agrupa los changes `done` sin release asignado,
calcula el bump semver correcto, actualiza `package.json`, crea el tag git
y dispara el publish en GitHub Actions — sin que el owner tenga que hacer
nada manualmente más allá de decidir cuándo soltar el release.

## Investigation

Convenciones de bump ya discutidas:

| Tipo de change | Bump |
|---|---|
| `fix`, `bug` | patch |
| `feature` | minor |
| breaking (flag en frontmatter) | major |
| `chore`, `audit`, `refactor` | ninguno |

Preguntas a resolver:
- ¿Dónde se persiste qué changes ya entraron en un release? (campo en frontmatter vs archivo de release)
- ¿El bump es explícito (`sl release minor`) o calculado automáticamente desde los types?
- ¿El comando solo prepara (bump + tag) o también empuja y crea el GitHub Release?

## Proposal

**Flujo propuesto:**

1. `sl release [--dry-run]`
   - Lista todos los changes `done` sin `release` en frontmatter
   - Calcula bump: si hay algún `feature` → minor; si solo `bug`/`fix` → patch; si hay `breaking: true` → major; si solo `chore`/etc → aborta (nada que publicar)
   - Muestra diff de `package.json` y lista de changes que entran
   - Con `--dry-run` solo imprime, no escribe

2. Sin `--dry-run`:
   - Escribe nueva version en `package.json`
   - Añade `release: "vX.Y.Z"` al frontmatter de cada change incluido
   - Crea commit `chore(release): vX.Y.Z`
   - Crea tag `vX.Y.Z`
   - Instrucciones para push + GitHub Release (no hace push automático)

**Alternativa rechazada:** bump explícito (`sl release minor`) — pierde trazabilidad de por qué se eligió ese bump.

## Specification

- CR1: `sl release --dry-run` lista changes elegibles y bump calculado sin escribir nada
- CR2: sin changes elegibles, el comando falla con mensaje claro
- CR3: bump se determina por el tipo más grave entre los changes elegibles
- CR4: `breaking: true` en frontmatter de cualquier change elegible → major
- CR5: `package.json` version se actualiza con el bump correcto
- CR6: cada change elegible recibe `release: "vX.Y.Z"` en su frontmatter
- CR7: se crea commit y tag `vX.Y.Z` en el repositorio git
- CR8: el comando no hace push ni crea el GitHub Release automáticamente
