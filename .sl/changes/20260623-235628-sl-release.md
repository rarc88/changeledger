## Request

Incorporar un flujo de releases basado en los changes terminados. Spec Ledger
debe decidir de forma trazable qué changes entran en una entrega y calcular la
siguiente versión SemVer, sin asumir dónde guarda la versión cada proyecto ni
cómo publica sus artefactos.

El agente que opera el repositorio consumirá el plan calculado y aplicará la
versión en las superficies propias de su tecnología: por ejemplo,
`package.json` en Node, `pubspec.yaml` en Flutter o varios manifests en un
monorepo. El mismo agente ejecutará las verificaciones y el flujo Git/release
definido por el proyecto.

## Investigation

La propuesta inicial actualizaba `package.json` y creaba commit y tag desde el
CLI. Eso resolvería el caso npm de este repositorio, pero acoplaría el contrato
general a Node y asumiría una estrategia Git que no todos los consumidores
comparten.

Inferir el impacto exclusivamente desde `type` también es insuficiente. El tipo
expresa la naturaleza del trabajo, no siempre su impacto público: un `refactor`
puede corregir comportamiento observable, un `feature` puede ser interno y los
repositorios pueden definir tipos propios. Los tipos son buenos defaults, pero
cada change necesita poder declarar una excepción explícita.

La pertenencia a releases debe conservarse como una relación auditable sin
reescribir muchos changes históricos. Un manifiesto por release ofrece una
fuente única de verdad y sigue funcionando cuando los changes ya están
archivados.

## Proposal

### Impacto efectivo

Aceptar el campo opcional de frontmatter
`release_impact: none|patch|minor|major`. Cuando falta, el CLI usa el default
configurado para el tipo en `.sl/config.yml`; una configuración sin default para
un tipo elegible es un error accionable, no una inferencia silenciosa.

Los defaults iniciales del template serán:

| Tipo | Impacto |
|---|---|
| `feature` | `minor` |
| `bug` | `patch` |
| `audit`, `refactor`, `chore` | `none` |

`major` representa una ruptura SemVer, también durante `0.x`; por ejemplo,
aplicado a `0.4.0` produce `1.0.0`. Una excepción se expresa cambiando
`release_impact`, de modo que la causa del cálculo permanece en el change.

### Historial de releases

Persistir manifiestos versionados bajo `.sl/releases/<version>.yml`, con al
menos `version`, `created` y la lista ordenada de ids de changes. Un change se
considera publicado si aparece en un manifiesto; no se duplica `released_in` en
su frontmatter.

`sl release init <current-version>` adopta el flujo en un repositorio existente:
crea un manifiesto baseline que incluye los changes ya terminados y evita que
sean propuestos para la siguiente entrega. Se rechaza si ya existe historial de
releases o si la versión no es SemVer válida.

### Comandos

`sl release plan [--json]`:

1. Lee la versión más reciente de los manifiestos.
2. Selecciona todos los changes `done` que no aparecen en ningún manifiesto,
   incluidos los archivados.
3. Resuelve el impacto de cada change y toma el mayor para calcular la siguiente
   versión.
4. Presenta versión actual, siguiente versión, impacto y changes incluidos, sin
   escribir archivos.

Si todos los changes pendientes tienen impacto `none`, informa que no hace
falta una release y termina correctamente sin proponer una versión. `--json`
expone un contrato estable para agentes y automatizaciones.

`sl release record <version>` vuelve a calcular el plan y crea atómicamente el
manifiesto solo cuando `<version>` coincide con la versión calculada. No edita
manifests tecnológicos, no ejecuta verificaciones y no crea commits, tags,
GitHub Releases, pushes ni publicaciones.

El flujo esperado del agente es:

```text
sl release plan --json
    → descubrir y actualizar las superficies de versión del proyecto
    → sl release record <version>
    → ejecutar los quality gates del proyecto
    → commit/tag/release según el contrato local
```

### Alternativas descartadas

- Actualizar `package.json` desde `sl`: acopla el producto a Node y no cubre
  Flutter, Rust, Gradle, Xcode ni monorepos.
- Crear commit, tag o GitHub Release desde `sl`: invade políticas operativas que
  pertenecen al repositorio consumidor.
- Guardar `release` en cada change: duplica la relación y obliga a mutar varios
  documentos históricos por entrega.
- Recibir el bump como argumento normal: permite contradecir los impacts sin
  dejar la justificación en los changes.

## Specification

### CR1 — Adopción de un historial existente
- **Given** un repositorio sin manifiestos de release y con una versión SemVer actual
- **When** el agente ejecuta `sl release init <current-version>`
- **Then** se crea un manifiesto baseline con esa versión y los changes `done` actuales
- **And** una segunda inicialización se rechaza sin modificar archivos

### CR2 — Selección de changes elegibles
- **Given** changes activos o archivados, algunos `done` y otros no terminados
- **And** manifiestos que ya incluyen parte de los changes terminados
- **When** se ejecuta `sl release plan`
- **Then** el plan incluye exactamente los changes `done` aún ausentes de los manifiestos

### CR3 — Cálculo configurable de impacto
- **Given** changes elegibles cuyos tipos tienen impacts por defecto
- **And** un change puede sobrescribir su default con `release_impact`
- **When** se calcula el plan
- **Then** el bump corresponde al mayor impacto efectivo
- **And** un tipo sin default ni override produce un error accionable

### CR4 — Cálculo estricto de SemVer
- **Given** una versión base válida y el mayor impacto efectivo del plan
- **When** se calcula la siguiente versión
- **Then** `patch`, `minor` y `major` aplican las reglas SemVer correspondientes
- **And** un impacto `major` desde `0.x` produce `1.0.0`

### CR5 — Plan sin efectos laterales
- **Given** uno o más changes elegibles con impacto publicable
- **When** se ejecuta `sl release plan` o `sl release plan --json`
- **Then** se informa la versión actual, la siguiente, el impacto y los changes incluidos
- **And** no se modifica ningún archivo ni estado externo

### CR6 — Entrega sin impacto publicable
- **Given** que todos los changes elegibles tienen impacto `none`
- **When** se ejecuta `sl release plan`
- **Then** el comando informa que no hace falta una release y termina correctamente
- **And** no propone ni registra una nueva versión

### CR7 — Registro consistente y atómico
- **Given** un plan que calcula una nueva versión
- **When** se ejecuta `sl release record <version>` con esa versión
- **Then** se crea atómicamente un manifiesto que contiene exactamente los changes del plan
- **And** una versión distinta, duplicada o inválida se rechaza sin escritura parcial

### CR8 — Frontera tecnológica y operativa
- **Given** un release planificado o registrado
- **When** finaliza cualquier comando `sl release`
- **Then** el CLI no modifica archivos de versión propios del stack
- **And** no crea commits, tags, releases remotas, pushes ni publicaciones

### CR9 — Validación del nuevo esquema
- **Given** configuración de impacts, manifests de release o `release_impact` en changes
- **When** se ejecuta `sl check`
- **Then** se validan enums, SemVer, ids existentes, duplicados y consistencia del historial
- **And** los errores identifican el archivo y el valor inválido
