## Request

Convertir el `README.md` raíz en una presentación centrada en quien descubre,
evalúa o instala `@rarc88/spec-ledger` desde npm o GitHub. Mantener un onboarding
rápido y la referencia de uso cotidiano, trasladando los detalles exclusivos de
mantenimiento del repositorio a `CONTRIBUTING.md`.

El README publicado por npm debe explicar primero el problema que resuelve Spec
Ledger, permitir probarlo en pocos minutos y dirigir a documentación más
profunda sin mezclar el flujo del consumidor con pnpm, Biome, hooks o desarrollo
mediante `pnpm link`.

## Proposal

Reorganizar el README público con esta jerarquía:

1. Propuesta de valor y captura visual breve del viewer cuando exista un asset
   estable.
2. Requisitos, instalación e inicio rápido.
3. Flujo cotidiano `init → new → approve → implement → validate → graduate`.
4. Modelo conceptual `changes` y `specs`, junto con el rol del viewer.
5. Comandos principales agrupados por tarea, sin convertir el README en la
   referencia exhaustiva del CLI.
6. Compatibilidad, seguridad, estado pre-1.0 y enlaces al contrato y a la guía
   de contribución.

Mover a `CONTRIBUTING.md` la configuración de pnpm/Biome, política interna de
dependencias, hooks, quality gates de desarrollo y `pnpm link --global`. Conservar
en el README solo la garantía pública relevante, por ejemplo las plataformas
probadas y el requisito de Node.

Reducir la sección detallada de Definition of Ready a una explicación orientada
al beneficio y enlazar el contrato para sus reglas completas. Mantener enlaces
relativos válidos tanto desde GitHub como desde la presentación renderizada por
npm.
